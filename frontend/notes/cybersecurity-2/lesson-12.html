Recon Analysis & Reporting

🎯 Objective: Analyze collected data and prepare a reconnaissance report

Data without analysis is just noise. A professional report transforms raw scan results into actionable intelligence for decision-makers.

📌 Why Reporting Matters:
  • Organizes chaotic scan data into clear findings
  • Communicates risks to non-technical stakeholders
  • Provides documentation for remediation teams
  • Serves as legal evidence if needed
  • Demonstrates professional competence

🔍 Understanding Recon Analysis

🤔 WHAT is Recon Analysis?

Recon analysis is the process of taking raw data from your OSINT and Nmap scans and turning it into meaningful, organized information. You're not just listing open ports — you're understanding what those ports mean for the target's security posture.

📋 What to Organize in Your Findings
  • Domain information — WHOIS data, registrar, name servers, registration/expiration dates
  • IP addresses — Target IP ranges, resolved IPs from DNS queries
  • Open ports — Which ports are accepting connections? (21, 22, 80, 443, 445, etc.)
  • Services running — Software names and version numbers (Apache 2.2.8, vsftpd 2.3.4, OpenSSH 4.7p1)
  • Potential vulnerabilities — Known CVEs for detected versions, weak configurations
  • Subdomains discovered — admin., mail., dev., test., api.
💡 How to Analyze Each Finding
  • Port 21 (FTP): Is anonymous login allowed? Is it outdated vsftpd 2.3.4 (has backdoor)?
  • Port 22 (SSH): Is it using weak key exchange? Allow password authentication? Old version vulnerable to CVE-2016-6210?
  • Port 80 (HTTP): What web server and version? Any directory listing? Admin panels exposed?
  • Port 443 (HTTPS): SSL/TLS version? Weak ciphers? Certificate expiry?
  • Port 445 (SMB): Vulnerable to EternalBlue (MS17-010)?
💡 Pro Tip: Always verify findings with multiple tools. Nmap might misidentify a service. Use nc -nv <IP> <port> to manually grab banners and confirm.

📄 Recon Report Structure (Part 1)

A professional reconnaissance report follows a standard structure that executives, technical teams, and legal departments can all understand.

1️⃣ Introduction

What to include:

  • Target description: What system or organization was assessed? (e.g., "Metasploitable 2 VM at 192.168.56.102")
  • Scope: What was included in the assessment? (e.g., "External reconnaissance only, no exploitation")
  • Authorization: Statement confirming permission to test
  • Assessment dates: When was the reconnaissance performed?
  • Classification: Confidential, Internal, Public
📋 Example:
"This report documents reconnaissance performed against the Metasploitable 2 lab environment on April 4, 2026. The assessment was authorized as part of Cybersecurity Level 2 training and was conducted within an isolated lab network."
2️⃣ Methodology

What to include:

  • Tools used: WHOIS, DIG, Nmap, etc. (include version numbers)
  • Techniques applied: Passive OSINT, active port scanning, service version detection, vulnerability scripts
  • Commands executed: Show the exact commands run
  • Scan parameters: Timing, port ranges, evasion techniques used
# Example methodology section:
1. WHOIS lookup via command line
2. DIG DNS enumeration (A, MX, NS, TXT records)
3. Nmap basic scan: nmap -sS 192.168.56.102
4. Nmap version scan: nmap -sV 192.168.56.102
5. Nmap vuln scripts: nmap --script vuln 192.168.56.102

📄 Recon Report Structure (Part 2)

3️⃣ Findings

Organize findings into clear categories:

  • Domain/Network Information: IP addresses, DNS records, WHOIS data
  • Open Ports Table: Port number, protocol, service, version
  • Service Details: Banner information, configuration notes
  • Vulnerability Findings: CVE numbers, severity, description
| Port | Service | Version | Status |
|------|----------|------------------|--------|
| 21 | FTP | vsftpd 2.3.4 | Open |
| 22 | SSH | OpenSSH 4.7p1 | Open |
| 80 | HTTP | Apache 2.2.8 | Open |
| 445 | SMB | Samba 3.0.20 | Open |
4️⃣ Risk Analysis & 5️⃣ Conclusion

Risk Analysis:

  • For each finding, explain the potential impact if exploited
  • Assign risk level (Critical/High/Medium/Low)
  • Provide remediation recommendations
📋 Example Risk Assessment:
"vsftpd 2.3.4 contains a known backdoor (CVE-2011-2523). This allows remote attackers to gain root access. Risk: CRITICAL. Recommendation: Upgrade vsftpd to version 2.3.5+."

Conclusion:

  • Summarize key findings (2-3 sentences)
  • State the overall security posture
  • List priority remediation actions

Here's what you might find when scanning the Metasploitable 2 VM. Use this as reference for your own analysis.

🔓 High Risk Findings
  • vsftpd 2.3.4 (port 21) — Contains backdoor allowing root access. CRITICAL
  • SMB 3.0.20 (port 445) — Vulnerable to EternalBlue/ms17-010. CRITICAL
  • Apache 2.2.8 (port 80) — Outdated, multiple vulnerabilities. HIGH
  • OpenSSH 4.7p1 (port 22) — Old version, various CVEs. MEDIUM
📋 Other Open Ports
  • Port 23: Telnet (unencrypted, sends passwords in plain text)
  • Port 25: SMTP (mail server, possible open relay)
  • Port 53: DNS (possible zone transfer)
  • Port 80: HTTP (Apache web server)
  • Port 443: HTTPS (self-signed certificate)
  • Port 3306: MySQL (default credentials?)
  • Port 5432: PostgreSQL
  • Port 5900: VNC (default password?)
💡 How to Document Vulnerabilities:
### Finding: vsftpd 2.3.4 Backdoor
- Port: 21/tcp
- Version: vsftpd 2.3.4
- Vulnerability: CVE-2011-2523 - Backdoor allows root access
- Risk: CRITICAL
- Remediation: Upgrade to vsftpd 2.3.5+ or remove service
