SSH & Log Analysis

Objective: Connect to remote systems and analyze logs

Remote access is a core skill. Log analysis turns raw data into threat intelligence.

# Every connection leaves a trace. Every trace tells a story.
$> ssh analyst@target && grep "Failed" /var/log/auth.log

🔧 Lab 1: SSH Connection

SSH (Secure Shell) is the standard protocol for encrypted remote administration. It replaces older, insecure protocols like Telnet and rlogin.

📡 Basic SSH Command
ssh user@target_ip

user = the account you want to connect as on the remote machine.
target_ip = IP address or hostname of the remote system.

🔐 What happens behind the scenes?
  • Client and server negotiate encryption algorithms
  • Server presents its host key (fingerprint)
  • Client verifies the key (first connection prompts for acceptance)
  • User authentication (password or SSH key)
  • Encrypted session established
📌 Expected Result
✅ Remote access to another machine
After successful authentication, you'll see the remote machine's shell prompt. Every command you type now runs on the remote system.
⚠️ First-time connection warning:
The authenticity of host '192.168.1.10' can't be established.
This is normal — verify the fingerprint out-of-band if security is critical.

🔧 Lab 2: Log Analysis

System logs are the silent witnesses of everything that happens on a Linux machine. /var/log/auth.log records authentication-related events.

📄 View login attempts
cat /var/log/auth.log

Shows the entire authentication log. Can be long — use less or tail for easier reading.

tail -50 /var/log/auth.log # last 50 lines
🔍 Search failed logins
grep "failed" /var/log/auth.log

grep filters lines containing the word "failed". Case-sensitive, so "Failed" vs "failed" matters — try both.

grep -i "failed" /var/log/auth.log # case-insensitive
👁️ What to Observe
  • Failed login attempts — entries with "Failed password" indicate authentication failures
  • Repeated login failures — many failures from the same IP suggests brute force
  • Unknown IP addresses — connections from unexpected geolocations or networks
  • Successful logins at odd hours — potential unauthorized access
  • Invalid users — attempts to log in as 'admin', 'root', 'test' (enumeration)

🔍 Digging Deeper into Logs

Real-world analysis requires combining commands to extract meaningful intelligence.

📊 Count failed attempts per IP
grep "Failed password" /var/log/auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -rn

This pipeline: filters failures → extracts IP addresses → counts unique occurrences → sorts by highest count.

📅 Failed logins by date
grep "Failed password" /var/log/auth.log | awk '{print $1, $2, $3}' | sort | uniq -c

Shows which days had the most authentication failures — helpful for identifying attack timing.

📂 Other important log files
  • /var/log/syslog — general system messages
  • /var/log/kern.log — kernel events (hardware, drivers, security modules)
  • /var/log/btmp — failed login attempts (binary format, use lastb command)
  • /var/log/wtmp — login history (use last command)

  • ✅ Ability to access remote systems — you can now securely connect to servers, VMs, and network devices using SSH.
  • ✅ Ability to detect suspicious login activity — you understand where authentication logs live and how to extract evidence of attacks.
  • ✅ Practical log analysis workflow — from viewing raw logs to filtering, counting, and identifying patterns.
🎯 Real-world application

These skills translate directly to:

  • Incident response — checking for unauthorized SSH access after a breach alert
  • Security monitoring — setting up log watches for brute force detection
  • Forensics — reconstructing attacker activity from authentication logs
  • System administration — troubleshooting access issues for legitimate users
